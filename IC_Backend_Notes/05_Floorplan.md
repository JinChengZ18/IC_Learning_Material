## 1. 认识 Floorplan：从逻辑网表到物理骨架

### 1.1 本讲主线：从逻辑网表到物理骨架

布图规划（floorplan）是数字后端从“逻辑世界”进入“物理世界”的第一层承诺。综合后的门级网表（gate-level netlist）只说明实例（instance）和网络（net），并不说明任何单元（cell）、宏单元（macro）、IO 或电源结构应该放在芯片的什么位置。Floorplan 的任务，就是把这张逻辑网表落成可实现的物理骨架。

这一层骨架至少包括：

- 晶粒（die）边界和核心区（core）；
- IO / 焊盘（pad）/ 凸点（bump）的大致位置；
- 硬宏（hard macro）的位置、朝向和固定状态；
- 标准单元可放置区（standard cell placement area）；
- 电源环（power ring）、电源条带（power stripe）、电源网格（power grid）和标准单元供电轨（standard-cell rail）；
- 阻挡（blockage）、留边（halo）、区域约束（region）等物理约束。

后续布局、CTS、布线与电源签核都会沿着这个骨架继续细化。Floorplan 阶段改动成本低，签核阶段再改动则可能牵一发而动全身。

### 1.2 本讲路线：按工程决策顺序展开

本章按工程决策顺序组织，先回答 Floorplan 的定位、目标、输入输出和设计导入（design import）前提；再进入晶粒 / 核心区尺寸、IO 环（IO ring）、利用率（utilization）、试布线（trial route）、宏单元、布局区域（placement region）和阻挡；随后讨论层次化实现中的时序预算（timing budget）、ILM、引脚分配（pin assignment）与穿通（feedthrough）；最后进入电源规划（power planning）、IR 压降（IR drop）、EM、电源网格创建，以及一条典型 Innovus 流程如何串起来。

理解这条主线后，遇到问题可以沿着同一条路径反查：

| 现象 | 优先反查 |
|------|----------|
| 布线拥塞 | 利用率、宏通道、引脚可达性、布线阻挡 |
| 时序变差 | 宏单元 / 引脚位置、层次化时序预算、长绕线 |
| IR 压降超标 | 电源焊盘、电源网格、过孔（via）、高功耗宏单元位置 |
| EM 风险 | 电流密度、金属宽度（metal width）、过孔阵列（via array）、高功耗集中区 |

### 1.3 Floorplan 的目标

Floorplanning 本质上是一个**映射**：

- 逻辑描述：netlist，描述“谁和谁相连”；
- 物理描述：floorplan，描述实例、IO、宏单元和电源网络在芯片上如何摆放、连接和供电。

这一步要同时服务三个目标：

1. 减小芯片面积；
2. 减小关键路径延迟；
3. 减小布线拥塞。

这三个目标**天然冲突**。面积压得太紧会拥塞，通道（channel）留得太宽会浪费面积，电源网格做得过强又会抢走信号布线资源。也就是说，Floorplan 是在面积、时序、拥塞和供电可靠性之间做**第一轮工程取舍**。

### 1.4 全芯片设计视角

从全芯片视角看，floorplan 需要同时考虑：

| 对象 | Floorplan 阶段要回答的问题 |
|------|----------------------------|
| 芯片尺寸 | 晶粒 / 模块有多大，长宽比（aspect ratio）如何 |
| 核心放置区 | 标准单元可放置区在哪里，利用率多少 |
| 对外接口 | IO 焊盘 / 凸点 / 封装引脚（package pin）如何对应 |
| Hard IP / Macros | SRAM、ROM、PLL、模拟 IP 等硬块放在哪里 |
| 电源传输 | 电源焊盘数量和位置，电源网格、环、条带怎么建 |
| 多电压 | 电源域（power domain）、电压岛（voltage island）、always-on 资源如何预留 |
| 时钟方案 | 时钟分布（clock distribution）是否需要特殊通道或宏单元位置配合 |
| 扁平 / 层次化 | 整颗芯片一次实现，还是切成多个模块（block）实现 |

**越早把这些结构决定清楚，修改成本越低**；拖到 CTS、布线或签核阶段才发现问题，返工成本会迅速上升。

### 1.5 输入、输出与设计导入前提

Floorplan 的输入至少需要：

| 输入 | 作用 |
|------|------|
| **设计网表（design netlist）** | 提供实例和连接关系 |
| **面积需求（area requirements）** | 估算核心区尺寸、利用率和宏单元空间 |
| **电源需求（power requirements）** | 决定电源焊盘、电源网格、高功耗宏单元位置 |
| **时序约束（timing constraints）** | 影响模块划分、引脚分配、宏单元数据流 |
| **物理分区（physical partitioning）** | 层次化设计中的模块边界和约束 |
| IO / 宏单元提示（可选） | 减少初期迭代成本 |

输出也必须具体到工具数据库可以继续使用：

- 晶粒或模块面积；
- IO 已摆放；
- 硬宏已摆放并固定；
- 电源网格初步设计完成；
- 必要的电源预布线（power pre-routing）已完成；
- 标准单元可放置区已定义；
- 阻挡、留边、区域约束等已落入数据库；
- 可布线性（routability）、时序和电源完整性（power integrity）已有早期评估。

## 2. 几何、可布线性与空间约束

### 2.1 IO 环与芯片尺寸

芯片几何可以先按**由外到内**理解：

```text
晶粒（Die） ⊃ IO / 焊盘环（Pad Ring） ⊃ 核心区（Core） ⊃（宏单元 + 标准单元行）
```

IO 引脚分配往往由前端、系统或封装侧先提出，但物理设计（physical design）必须参与评审。原因有两个：

1. I/O 不像逻辑晶体管那样随工艺快速缩小，IO 单元（IO cell）和焊盘面积非常贵；
2. I/O 不只传信号，还负责供电，电源焊盘（power pad）和地焊盘（ground pad）的数量、位置会直接影响 IR 压降和 EM。

选择芯片尺寸时，先判断设计是核心受限（core limited）还是焊盘受限（pad limited）：

| 类型 | 尺寸由谁决定 | 优化重点 |
|------|--------------|----------|
| 核心受限 | 逻辑规模、宏单元、布线资源决定芯片大小 | 利用率、宏单元摆放、布线资源 |
| 焊盘受限 | IO 数量、焊盘间距（pad pitch）、封装引脚分配决定晶粒不能再小 | IO 环、焊盘 / 凸点分配、封装协同 |

### 2.2 利用率（Utilization）与试布线（Trial Route）

Floorplan 阶段的利用率通常指**标准单元占核心区面积的比例**，常见初值可能是 70%。

利用率太高会导致：

- 布线拥塞上升；
- 布局合法化（placement legalization）和优化自由度不足；
- 引脚密集单元（pin-dense cell）周围局部拥塞；
- 电源网格与信号布线抢资源；
- 时序优化难以插入缓冲器（buffer）或调整位置。

利用率太低也不是好事，会浪费晶粒 / 模块面积，并拉长平均互连。

工程上还要区分两个口径：

```text
总利用率（Total Util）= (Σ标准单元面积 + Σ宏单元面积) / 核心区面积
有效利用率（Effective Util）= Σ标准单元面积 / (核心区面积 - 阻挡 - 宏单元留边 - 禁布区 - ...)
```

不能只凭利用率决定晶粒尺寸。完成初始面积估算后，需要跑快速试布线或全局布线（global routing）估计，检查拥塞，再决定是否放大核心区、调整综合结果、移动宏单元、修改引脚分配或重新分配布线资源。

### 2.3 网表唯一化（Netlist Uniquify）

进入物理域前，网表必须**唯一化**。唯一化网表的意思是每个子模块只被引用一次。

为什么这件事和 floorplan 有关？因为物理优化要按实例独立移动、插缓冲器、优化。如果两个实例共享同一个模块定义（module definition），工具想优化 `m1/u1` 时，就可能同时改变 `m2/u1`。

一个简化例子：

```text
top
  m1: amod
    u1: BUFFD1
  m2: amod
    u1: BUFFD1
```

不做唯一化时，`m1/u1` 和 `m2/u1` 的物理优化边界不够清晰。综合后的网表必须在布局之前完成唯一化，可以由综合工具完成，也可以在设计导入阶段完成。

### 2.4 硬宏摆放（Hard Macro Placement）

硬宏包括 SRAM、ROM、PLL、模拟 IP、第三方 hard IP 等。它们面积大、形状固定、引脚固定、内部不可被摆放器（placer）任意拆开，对布线、时序和电源都有强影响。

宏单元摆放的主线是：**把大宏推到边缘或角落**，让标准单元区域保持一个尽量完整的大矩形。

常用原则：

1. 沿边 / 沿角：大宏尽量靠核心区边界或角落，不在中心挖洞；
2. 按数据流（dataflow）就近：让强连接的宏单元 / 逻辑靠近，用飞线（flyline）看连接权重；
3. 保留布线通道：宏单元与宏单元、宏单元与边界之间要留足信号、电源、时钟通道；
4. 改善引脚可达性（pin accessibility）：必要时旋转宏单元，让引脚面向更开放的区域；
5. 避免引脚对着窄通道或角落；
6. 摆好后固定：硬宏位置确定后通常标记为固定。

对引线键合（wire-bond）设计，高功耗宏单元不宜放在芯片中心太深的位置，因为电源焊盘通常在边缘，供电路径越长，IR 压降压力越大。

### 2.5 布局区域（Placement Region）：表达放置意图

布局区域是帮助布局布线工具（P&R tool）表达设计意图的方式，用来引导或限制某些单元的摆放区域。

| 类型 | 含义 | 强度 |
|------|------|------|
| 软引导（soft guide） | 希望这些单元聚在一起，但无固定区域 | 最软 |
| 引导区域（guide） | 尽量放在指定区域 | 较软 |
| 区域约束（region） | 指定单元必须放在该区域，其他单元也可进入 | 较硬 |
| 围栏约束（fence） | 指定单元必须放在该区域，其他单元不可进入 | 最硬 |

越硬的约束越容易制造局部拥塞，因此区域约束 / 围栏约束应当**只用于必要的结构性意图**，而不是用来“替工具做布局”。

### 2.6 布局阻挡（Placement Blockage）与留边（Halo）

布局阻挡用于**限制标准单元摆放**：

| 类型 | 作用 |
|------|------|
| **硬阻挡（hard blockage）** | 完全禁止放置标准单元 |
| **软阻挡（soft blockage）** | 初始布局阶段禁用，优化阶段可用于缓冲器 / 反相器 |
| **部分阻挡（partial blockage）** | 限制区域最大密度，例如最多 40% |

留边（halo），也叫 padding / keepout margin，是宏单元外围保留的一圈空白区域。它主要用于改善宏引脚可达性，给缓冲器 / 反相器插入、电源布线和信号布线留近宏通道。

高频区别：

```text
留边（halo）   -> 跟随宏单元移动
阻挡（blockage）-> 多为固定坐标区域
```

### 2.7 布线阻挡（Routing Blockage）与好 Floorplan

布线阻挡**限制的是走线，而不是单元摆放**。它可以指定某个布线层（routing layer）或层范围，例如禁止 M1-M3 在某个区域走线。

常见用途包括：

- 保护宏单元上方特殊区域；
- 给电源结构预留资源；
- 避免低层金属被不合适的信号占用；
- 保护噪声敏感模拟区。

好的 floorplan 有几个图上就能看出来的特征：

- 保持单个大的核心放置区；
- RAM 这类大块尽量放在角落或边缘；
- 保留宽布线通道；
- 避免收缩狭窄通道；
- 不让大量引脚对着窄通道；
- 必要时旋转宏单元改善引脚可达性；
- 引脚不挤在模块角落；
- 用阻挡 / 留边提前保护可布线性。

## 3. 层次化接口与时序预算

### 3.1 扁平设计与层次化设计

本章承接上一章的几何与空间约束：当设计规模继续膨胀，**扁平流程（flat flow）**一次跑完整颗芯片会遇到运行时间（runtime）和内存压力。应对办法是**层次化设计（hierarchical design）**：把全芯片切成多个模块，每个模块独立做布局布线（P&R），再在顶层做全芯片时序和验证。

优点：

- EDA 运行时间更短；
- 内存更少；
- ECO 周期更快；
- 有利于设计复用。

代价（也正是后两节要解决的）：

- **全芯片时序收敛**更难；
- **引脚分配和穿通**更关键；
- **时序约束预算**更复杂；
- **模块抽象 / ILM / ETM** 需要维护。

换句话说，层次化不会让问题消失，它只是把「大问题」拆成更清楚的「边界规划问题」——这正是接下来时序预算与引脚 / 穿通要回答的。

### 3.2 时序预算（Timing Budget）与 ILM

层次化把芯片切成模块后，第一个要回答的边界问题就是时序：**芯片级约束（chip-level constraints）必须映射成模块级约束（block-level constraints）**，否则各模块根本不知道自己边界上该满足什么。例如顶层某个输入到模块的路径预算为 1.5 ns，模块实现时就需要在模块边界建立对应的输入延迟（input delay）、输出延迟（output delay）、时钟不确定性（clock uncertainty）、负载（load）和驱动单元（driving cell）等约束。

```tcl
create_clock -name clk -period 2.0 [get_ports clk]
set_input_delay  0.6 -clock clk [get_ports data_in]
set_output_delay 0.7 -clock clk [get_ports data_out]
set_clock_uncertainty 0.10 [get_clocks clk]
set_driving_cell -lib_cell BUFX4 [get_ports data_in]
set_load 0.05 [get_ports data_out]
```

**预算合理**，各模块独立收敛后全芯片才容易收敛；**预算不合理**，单个模块看似通过，全芯片仍然可能失败。

正因为预算要在模块边界上反复核对，模块的内部细节就成了负担，于是有了抽象模型：**接口逻辑模型（Interface Logic Model，ILM）**保留模块边界附近与接口时序相关的逻辑，同时隐藏模块内部大量细节。抽取时序模型（Extracted Timing Model，ETM）也用于提供模块的抽象时序模型。它们的目的都是让全芯片时序分析**更快、更可控**。

### 3.3 引脚分配（Pin Assignment）与穿通（Feedthrough）

时序预算划好之后，模块还要真正连起来——引脚分配就是给边界「接口」找落点。顶层 IO 摆放决定芯片对外接口位置；模块级引脚分配决定层次化模块之间怎么连接。**两者都属于 floorplan，但影响点不同**：

- **顶层 IO** 影响封装、ESD、电源焊盘、芯片边界；
- **模块引脚** 影响模块间布线、时序预算、feedthrough 和全芯片收敛。

模块级引脚的约束常包括布线层、引脚间距、引脚尺寸、重叠规则、网络分组、引脚引导区域（pin guide）、数据流方向和试布线边界穿越情况。

引脚分配之所以连着穿通，是因为模块一旦挡在信号路径上，连接就走不直了：在无通道（channel-less）或通道资源紧张的设计中，**穿通（feedthrough）很关键**。若信号从分区 A 到分区 C，中间隔着分区 B，而 B 没有 feedthrough，信号可能被迫绕过整个模块。解决办法是在 B 上生成 feedthrough 引脚 / feedthrough 网络，把跨越 B 的连接拆成几段。

## 4. 电源规划与完整性

### 4.1 电源规划（Power Planning）：功耗与可靠性

电源规划要解决的**不只是“连上 VDD/GND”**。它同时关系到动态功耗（dynamic power）、静态漏电（static leakage）、IR 压降、电压跌落（voltage drop）、电迁移（electromigration）和自热磨损（self-heating wearout）。

**电源分布网络（Power Distribution Network，PDN）**需要：

- 把电流从焊盘 / 凸点送到晶体管；
- 保持稳定、低噪声的电压；
- 提供平均和峰值功耗需求；
- 为信号提供返回路径；
- 避免电迁移和自热磨损；
- 合理占用芯片面积和布线资源。

电源规划**从 floorplan 阶段就开始**，而不是布线后的补救。

### 4.2 IR 压降（IR Drop）

IR 压降是供电线上的电压下降：

```text
ΔV = I · R
```

芯片上的电源网格是一个巨大网络，工具会建立电阻矩阵，结合每个门的平均电流，求解每个节点的电压。若实际电压**低于容差下限**，单元延迟会增加，时序可能失败，严重时功能错误。

一个典型估算能说明问题：1 mm 长、100 nm 宽的 M1 电源轨，若方块电阻（sheet resistance）为 0.1 ohm/square，线电阻可达约 1000 ohm；若承载约 0.1 mA，IR 压降可达约 100 mV。这说明**细而长的低层电源线无法单独承担供电**，必须用更宽、更厚、更高层、多路径、多过孔的结构。

### 4.3 电迁移（Electromigration，EM）

**电迁移（EM）**是**电流长期流过导体时，电子动量推动金属原子迁移**，可能导致：

- **开路（open）**：单根线上形成空洞（void）；
- **短路（short）**：相邻线之间桥接（bridging）；
- **RC 改变**：即使没开短路，也可能造成性能退化。

IR 更偏**“瞬时电压够不够”**，EM 更偏**“长期电流密度能不能扛住”**。两者都与 floorplan 有关，因为电源焊盘、高功耗宏单元位置、网格宽度 / 间距 / 过孔都是在早期 floorplan 里决定或预留的。

### 4.4 电源分布网络（PDN）的基本取舍

电源线越强，IR / EM 风险越低，但信号布线资源越紧张。

| 选择 | 好处 | 代价 |
|------|------|------|
| 更宽电源线 | 降低电阻与 IR 压降 | 占用布线轨道 |
| 更多电源条带 | 提供更多电流路径 | 增加信号拥塞 |
| 更多过孔阵列 | 降低垂直连接电阻与 EM 风险 | 占用局部布线资源 |
| 更靠近电源焊盘 | 缩短供电路径 | 可能牺牲数据流 |

一个好的 **PDN** 要让电源轨尽量宽、尽量厚、尽量多路径，但**不能把信号布线资源全部吃光**。

### 4.5 电源/地布线（P/G Routing）方式

常见电源结构自上而下包括：

| 结构 | 作用 |
|------|------|
| 电源焊盘 / 凸点（power pads / bumps） | 从封装引入 VDD/VSS |
| 电源环（power ring） | 围绕芯片 / 核心区 / hard IP 的主干环 |
| 电源条带（power stripe） | 横跨核心区分担电流 |
| 电源网格（power mesh） | 纵横条带互连，降低等效电阻 |
| 标准单元供电轨（std-cell rail） | 沿标准单元行连接每个单元的 VDD/VSS |
| 专用布线（special route） | 连接焊盘、环、条带、模块引脚、核心区引脚和 follow-pin rail，并打过孔 |

标准做法是用**上层厚金属做网格或条带，下层连接到标准单元 follow-pin rail**，中间用多重过孔和过孔堆栈（via stack）连起来。

### 4.6 电源网格创建（Power Grid Creation）

电源网格创建的本质，是**在 IR 压降、EM 和布线资源之间取舍**。开始之前，需要功耗预算和初始功耗估算，知道平均电流、最大电流密度，再决定总体网格结构。

需要决定的参数包括：

- 总体网格结构；
- 是否有电源门控（power gating）或多电压；
- 每个电压域（voltage domain）的电源焊盘数量和位置；
- 使用哪些金属层；
- 电源条带的宽度 / 间距；
- 过孔堆栈如何放、会占掉多少布线轨道；
- 要不要电源环；
- 层次化模块是否需要屏蔽（shielding）；
- 初始电源网络分析。

参考 Innovus 命令骨架：

```tcl
globalNetConnect VDD -type pgpin -pin VDD -all
globalNetConnect VSS -type pgpin -pin VSS -all

addRing   -nets {VDD VSS} -type core_rings \
          -layer {top M9 bottom M9 left M8 right M8} -width 3

addStripe -nets {VDD VSS} -layer M9 -direction vertical \
          -width 2 -set_to_set_distance 40

sroute    -nets {VDD VSS} \
          -connect {corePin floatingStripe blockPin padPin}
```

### 4.7 电源网格与宏单元摆放

高性能、高功耗模块的宏单元摆放不能只看数据流，还要看供电路径。

实用规则：

- 高功耗模块靠近边界电源焊盘；
- 高功耗模块之间不要过度集中；
- 宏通道不仅给信号布线，也要给电源条带、过孔、时钟预留空间；
- 电源焊盘、宏单元位置和网格密度必须一起迭代。

IR 压降通常用颜色图（color map）找热点（hot spot）。修复不一定是全局加密，有时只需要在热点附近加一条电源线、增加过孔阵列、加宽电源条带、移动高功耗宏单元，或增加 / 重分配电源焊盘。

## 5. 工具流程与调试收束

### 5.1 Innovus 流程：从初始化到阻挡

实践中可以按以下工具无关步骤串起 floorplan 流程：

```text
1. 初始化设计（Init Design）
   - 读入网表 / LEF / MMMC / SDC / IO 摆放信息

2. 指定布图规划（Specify Floorplan）
   - 布图尺寸
   - 长宽比
   - 目标利用率

3. 摆放硬宏（Place Hard Macros）
   - 绝对坐标或相对位置摆放
   - 标记为固定
   - 定义留边和阻挡

4. 定义区域与阻挡（Regions & Blockages）
   - 布局区域
   - 布局阻挡
   - 布线阻挡

5. 定义全局电源网络（Define Global Nets）
   - 将顶层 VDD/GND 名称映射到 IP 的电源/地引脚

6. 创建电源环（Create Power Rings）
   - 芯片外围电源环
   - 必要时为 hard IP 创建电源环

7. 建立电源网格（Build Power Grid）
   - 标准单元 follow-pin 供电轨
   - 电源条带 / 网格
   - 可靠连接 hard IP 电源

8. 分配引脚（Assign Pins）
   - 层次化实现中的模块边界引脚
```

从而形成反复调整的闭环。

### 5.2 本讲收束：判断一个好 Floorplan

一个好 floorplan 的判断标准很简单：让后面的布局、CTS、布线和电源签核**都有路可走**。

| 问题 | 反查方向 | 常见动作 |
|------|----------|----------|
| 布线绕不出 | 利用率、宏通道、引脚可达性 | 降利用率、拓宽通道、调引脚、改阻挡 |
| 时序收不回 | 宏单元 / 引脚位置、时序预算 | 按数据流重摆宏单元，重新预算接口路径 |
| IR 压降超标 | 电源焊盘、网格、过孔、高功耗宏单元 | 加密网格、增加过孔、移动高功耗宏单元 |
| EM 违例 | 电流密度、金属宽度、过孔阵列 | 加宽金属、并联过孔、分散高功耗模块 |
| PG 未连通 | 全局电源名、宏单元电源/地引脚、专用布线 | 校正 VDD/GND 命名并补专用布线 |

Floorplan 的工作方式可以总结为：

```text
布图规划（floorplan）
  -> 试布局 / 试布线
  -> 拥塞 / 时序 / IR 早期分析
  -> 调整利用率 / 宏单元 / 通道 / 引脚 / PG
  -> 重复直到可接受
  -> 详细布局
```

后续进入布局时，不再重新回答“芯片骨架是什么”，而是在这个骨架上把标准单元放到合法、可优化、可收敛的位置。

## 6. 布图规划算法（Floorplanning Algorithms）

前几章把 floorplan 当作一套**工程决策**来讲：怎样定尺寸、摆宏单元、留通道、建电源网格。这一章换一个视角，看工具内部到底**用什么算法**把一堆模块摆成一个布图。理解算法不是为了手算，而是为了知道工具在优化什么、什么时候会卡住、为什么固定轮廓（fixed-outline）下还会“面积够却塞不进”。

学术上的布图规划问题可以这样描述：给定一组矩形模块（module / block），每个模块有面积，可能还带形状约束（长宽比范围），要为每个模块定一个不重叠的位置，使某个代价最小。代价通常是**面积（area）+ 线长（wirelength）+ 长宽比（aspect ratio）**的组合——这与第 1 章“面积 / 延迟 / 拥塞三者平衡”是同一件事，只是换成可优化的目标函数。

经典做法把布图规划拆成两步：

- __拓扑生成（topology generation）__：只定模块之间的**相对位置**（谁在谁左边 / 上边），不管确切形状，必要时只用面积估算。
- __尺寸确定（sizing）__：在给定拓扑下，为每个模块选具体的宽高（形状），让整体面积 / 线长最优。

模块按形状自由度分两类：

- __硬模块（hard module）__：形状、尺寸固定（如 SRAM、第三方 IP），最多只能旋转 / 翻转；
- __软模块（soft module）__：面积固定但长宽比可在一定范围内变形（如标准单元聚成的逻辑块）。

> 本章按**发展脉络**先梳理五个阶段，再分点讲透几个 landmark 算法：切片 + 模拟退火、非切片表示（Sequence Pair / B\*-tree / TCG）、固定轮廓与现代感知式布图、工业布图与原型。其余表示法和变体进对比表或一句话带过。内容忠于《Handbook of Algorithms for Physical Design Automation》（Ch.8–13），存疑处会标注。

### 6.1 发展脉络

布图规划算法大致经历了五个阶段，每一阶段都是为了解决前一阶段“表达不了”或“不贴近工业”的问题：

| 阶段 | 时期 / 代表 | 核心思想 | 解决了什么 / 仍有什么问题 |
|------|------------|----------|---------------------------|
| ① 早期构造法 | Ch.8，1980s | 拓扑生成 + sizing 两步；切片嵌入、min-cut 划分、**矩形对偶（rectangular dualization）**、层次枚举（BEAR） | 把“相对位置”形式化；但多数构造法只生成**可切片（slicible）**布图，表达力有限 |
| ② 切片 + 模拟退火 | Ch.9，Wong–Liu 1986 | 切片树（slicing tree）+ **Polish 表达式**编码；Stockmeyer 形状曲线做 sizing；模拟退火（SA）搜索 | 编码紧凑、sizing 有最优算法；但切片结构**无法表达所有 packing**，硬模块利用率可能不佳 |
| ③ 非切片表示 | Ch.10–11，约 1995–2005 | 镶嵌（mosaic）表示（CBL 等）与 packing 表示（O-tree、B\*-tree、**Sequence Pair**、**TCG** 等） | 能表达任意紧致布图，硬模块打包更紧；表示众多，各有解空间 / 复杂度 / 透明性权衡 |
| ④ 现代布图规划 | Ch.12，2000s | **固定轮廓（fixed-outline）**+ slack 引导；解析 / 数学规划；whitespace 当资源；拥塞 / 缓冲 / 时序-吞吐 / 功耗 aware | 目标从“最小面积”转为“塞进给定轮廓 + 多目标”；更贴近真实流程 |
| ⑤ 工业布图与原型 | Ch.13 | floorplanning（早期诊断）vs prototyping（后期可行性闸门）；flat vs hierarchical；pin assignment + timing budgeting；routability | 容忍不完整数据、欢迎人工介入；block 主导、层次化预算切块 |

一句话串起来：**从“能不能形式化表达布图”，到“能不能高效搜索 + 最优 sizing”，再到“能不能表达任意布图”，最后到“能不能贴近工业真实约束”**。前三阶段是学术上把问题越做越通用、越做越快；后两阶段是把目标从单纯面积拉回到工业关心的时序、布线、功耗与可行性。

下面分点讲透每个阶段里的关键算法。

### 6.2 切片式布图与模拟退火

切片式布图是历史上第一类被彻底算法化的布图。**可切片布图（slicing floorplan）**指：整块矩形可以用一条横线或竖线递归地一分为二，直到每个小矩形恰好放一个模块。它的好处是结构规整、搜索空间小、软模块形状可被充分利用以紧密打包，因此算法跑得快。

__切片树（slicing tree）与 Polish 表达式__——切片结构可以用一棵有序二叉树表示：每个内部节点标 **V**（竖切）或 **H**（横切），每个叶子是一个模块。对切片树做**后序遍历（postorder）**就得到 **Polish 表达式**（逆波兰式），长度为 2n−1（n 个模块、n−1 个割）。例如某切片树的 Polish 串是 `412H3VV`。

问题是同一个切片布图可能对应**多棵**切片树（连续的同向割可以换顺序），于是同一布图有多个 Polish 串，搜索空间被无谓放大、还分布不均。解决办法是**规范化（normalized）**：只用**斜树（skewed tree）**——任何节点与其右孩子不同标号。斜树对应的 Polish 表达式里**没有连续的 V 或连续的 H**，且长度 2n−1 的规范 Polish 表达式与切片结构**一一对应**。这就是 Wong–Liu 把规范 Polish 表达式选作搜索状态的原因：**一个状态恰好对应一个切片结构，不重不漏**。

__Stockmeyer 形状曲线 DP（sizing）__——给定切片树，怎样为每个模块选形状（软模块可变形、硬模块可旋转）使总面积最小？Stockmeyer 给出一个自底向上的动态规划。核心是**形状曲线（shape curve）**：把一个模块（或子布图）所有不冗余的 (宽, 高) 选择，按宽递减、高递增排成一条 Pareto 阶梯。合并规则是：

- 内部节点是 **H 割（横切，子块上下叠放）**：高相加、宽取较大者 → `(max(wᵢ,wⱼ), hᵢ+hⱼ)`，即沿 **y 方向**相加；
- 内部节点是 **V 割（竖切，子块左右并排）**：宽相加、高取较大者，即沿 **x 方向**相加。

合并两条已排序的阶梯曲线像归并两个有序表，丢掉被支配的冗余点，子曲线长度之和线性增长，整棵树做完是 O(n²)。到根节点后，在根的形状曲线上取面积最小（且满足芯片长宽比约束）的一点，再顺指针自顶向下定下每个模块的形状。**Shi 改进**用平衡二叉树存形状曲线，把复杂度降到 O(m log m)（m 为所有模块形状总数），并证明这是面积最小化的**最优下界**。Otten / Zimmermann 把同一思想推广到分段线性曲线与未定向切片树（同时算 H、V 两条曲线再取下包络）。

__Wong–Liu 模拟退火（SA）__——有了编码（规范 Polish）与 sizing（形状曲线），还需要一个搜索引擎来同时优化面积和线长。Wong–Liu（1986）用**模拟退火**，状态空间就是所有规范 Polish 表达式，靠三种**移动（move）**从一个状态走到相邻状态：

- __M1（换相邻模块）__：交换表达式里相邻的两个模块名；
- __M2（补一段割链 V↔H）__：把一串连续的割（一条 chain of cuts）整体取补（V 变 H、H 变 V）；
- __M3（换相邻的模块与割）__：交换相邻的一个模块和一个割。

M1、M2 必然仍是规范 Polish；M3 可能破坏规范性，所以反复挑选直到交换后仍规范为止。三种移动合起来保证任意规范表达式都能在有限步内互达（搜索空间连通）。代价函数是 **A + λ·W**（A=最小面积布图的面积，W=总线长，λ 控制二者权重）。每次移动只改变切片树的一小部分，所以最小面积布图可以**增量重算**——只在被改动的子树上重跑形状曲线合并，这是 SA 内层能跑得动的关键。

> 局限：切片结构**表达不了所有紧致布图**。对全是硬模块的情形，最优切片布图可能留下用不掉的空白。这正是下一节非切片表示要解决的问题。（注：理论上，切片树若再加一步压缩，可生成所有“极大紧致 / admissible 布局”，但实践中仍倾向用更直接的非切片表示。）

### 6.3 非切片表示

为了表达切片结构表达不了的布图，研究者提出了一批**非切片表示**。按能表达的布图结构，大致分两类：

- __镶嵌表示（mosaic）__：把矩形区域剖分成“房间（room）”，每个房间恰好一个模块、不留空房，内部只允许 T 形交叉。代表是**角块列表（Corner Block List，CBL）**，还有 Q-sequence、孪生二叉树 / 序列（TBT/TBS，后者编码非冗余）。镶嵌比切片更通用，但仍不能表达带空房的一般 packing。
- __打包表示（packing）__：最通用，能表达带空房的任意布图。又分**紧致型（compacted）**——O-tree、B\*-tree、Corner Sequence，只编码部分拓扑、打包时需用模块尺寸压到左下角；和**通用型（general）**——Sequence Pair、BSG、TCG、TCG-S、ACG，能完整编码任意两模块的几何关系。

下面讲透三个最有代表性的：Sequence Pair、B\*-tree、TCG。

__Sequence Pair（序列对，SP）__——Murata 等提出，用**一对模块名序列 (Π⁺, Π⁻)** 表达任意布图。两个序列里的先后顺序联合决定任意两模块的相对关系：

- 若 bᵢ 在 Π⁺ 和 Π⁻ 中**都**排在 bⱼ 前 → bᵢ 在 bⱼ **左**边；都排在后 → 右边；
- 若 bᵢ 在 Π⁺ 中排在 bⱼ 后、在 Π⁻ 中排在前 → bᵢ 在 bⱼ **上**方（反之下方）。

任意两模块要么水平关系、要么垂直关系，所以不会重叠。从 SP 得坐标：按“左于”关系建**水平约束图（horizontal constraint graph）**——节点是模块、权重是宽，源 s 到汇 t，bᵢ 左于 bⱼ 就连边 (i,j)；同理按“下于”关系建**垂直约束图**、权重用高。在两张图上各跑一次**最长路（longest path）**：模块的 x 坐标 = 水平图中从 s 到该点的最长路长，y 坐标同理；芯片宽 / 高 = s 到 t 的最长路长。最长路 O(n²)；用最长公共子序列（LCS）可加速到 O(n log log n)。解空间是 **(n!)²**（两个序列各 n! 种排列）。

__B\*-tree__——Chang 等提出，用**单棵有序二叉树**直接编码一个**紧致布局（admissible placement，左下都压实、不能再左移或下移）**，而且**布局与 B\*-tree 一一对应**。规则极简洁：

- __根__ = 左下角的模块；
- __左孩子__ = 在父模块**右邻且紧贴**的模块，几何上 `xⱼ = xᵢ + wᵢ`（同一行往右）；
- __右孩子__ = 与父模块**同一 x、在其上方**的模块，几何上 `xⱼ = xᵢ`（往上叠）。

从 B\*-tree 还原布局：DFS 遍历，x 坐标由上面两条规则一次定出；y 坐标用**轮廓线（contour）**数据结构（一条记录当前上边界的双向链表）边插边维护，使每个模块的 y 摊还 O(1)，整体打包 O(n) 线性时间。扰动靠三种操作：旋转模块、移动模块（删一个节点再插到别处）、交换两模块。**B\*-tree 与 O-tree 等价**，但用单棵二叉树、数据结构更简单、search/insert 是常数时间，且更易加各种约束，因此更常用（固定轮廓工具 Parquet 后来也加了 B\*-tree 选项）。

__TCG（传递闭包图，Transitive Closure Graph）__——Lin–Chang 提出，用**两张图**显式表达水平 / 垂直关系：水平闭包图 Cₕ（节点权=宽）放所有“左于”关系，垂直闭包图 Cᵥ（节点权=高）放所有“下于”关系。可行 TCG 满足三性质：两图都无环；每对模块恰好由其中一张图的一条边相连（保证不重叠）；每张图的传递闭包等于自身（消冗余）。从 TCG 打包同样是在两图上跑最长路得坐标，O(m²)。TCG 的卖点是**关系透明（transparent）**：模块间的几何关系直接写在图里，扰动操作（reverse 反转一条 reduction edge、move 把边从一张图挪到另一张）在**打包之前**就能知道会怎样改变几何关系，有利于更快收敛。**TCG-S** = TCG + 一个打包序列，借 SP 的序列把打包降到 O(m log m)。

__表示法对比__——

| 表示 | 类别 | 解空间 | 打包复杂度 | 通用性 | 关系透明性 |
|------|------|--------|------------|--------|------------|
| __切片 / 规范 Polish__ | 切片 | 与切片结构一一对应 | O(n)（形状曲线）| 仅可切片布图 | 否 |
| __CBL__ | 镶嵌 | O(n!·2³ⁿ⁻³) | O(n) | 镶嵌（无空房）| 部分 |
| __O-tree__ | 紧致 packing | O(n!·2²ⁿ/n¹·⁵) | O(n) | 紧致布局 | 否（需尺寸还原）|
| __B\*-tree__ | 紧致 packing | O(n!·2²ⁿ/n¹·⁵)（与 O-tree 等价）| O(n) 线性 | 紧致布局 | 否（需尺寸还原）|
| __Sequence Pair__ | 通用 packing | (n!)² | O(n²)，LCS 可 O(n log log n) | 任意布图 | 否（需打包后才知）|
| __BSG__ | 通用 packing | O(n!·C(n²,n)) | O(n²) | 任意布图 | 否 |
| __TCG__ | 通用 packing | (n!)² | O(n²) | 任意布图 | __是__ |
| __TCG-S__ | 通用 packing | (n!)² | O(n log n) | 任意布图 | __是__ |

> 两组等价关系值得记住：**B\*-tree ≡ O-tree**（如规范 Polish 之于斜切片树）；**SP ↔ TCG 等价**，可互相转换。它们表达力相同，区别在于邻域结构、操作效率与对约束的友好程度——这决定了在具体 SA 流程里选谁。

### 6.4 现代布图规划

进入 2000 年代，学术界开始反思经典假设与工业实际脱节（Kahng 2000 首先质疑）。最大的转变是问题**重新表述**：从“最小面积优化问题”变成“塞进给定轮廓的可行性问题 + 多目标”。

__固定轮廓 floorplanning（fixed-outline）+ slack__——经典布图是**无轮廓（outline-free）**的：用 SA 在长宽比约束下最小化面积。但真实流程里**先定 die 尺寸与封装、再做 floorplan**，所以现代表述是**固定轮廓**：边界宽高是**输入约束**，要把模块塞进这个矩形，而不是把面积压到最小。难点在于这是个**二维**约束——不是“面积够就行”，而是宽和高**都**要满足；一个 90 面积的解若是 15×6，仍然塞不进 10×10。

Adya–Markov 的关键想法是 **slack 引导的移动**。对一个 Sequence Pair，把模块往左下压得一组坐标、往右上压又得一组，两者之差就是该模块的 **slack**（x、y 各一个），衡量它在不破坏 SP、不增大芯片的前提下可挪动的余地。**零 slack 的模块在关键路径上**——芯片的某个维度只有移走零-slack 块才可能缩小；策略就是**把零-slack 块挪到高-slack 块旁边**（那里更可能有空间）。代价函数对越界量加惩罚。这套 slack-based move 比只惩罚越界的做法成功率高得多。工具 **Parquet** 实现了它（用 SP，后续版本加了 B\*-tree；经验上 B\*-tree 打包更紧、100 块以上更快）。Chen–Chang 用 B\*-tree + **Fast-SA**（高温随机、伪贪心局部、爬山三阶段退火）进一步加速。

__解析 / 数学规划__——除了 SA，也有把布图建成**网络流 / 数学规划**求解的：用 min-cost max-flow 判可行并分配面积（增量 / 交互式布图）、用混合整数（非）线性规划（MILP / MINP）做带目标的摆放。这类方法在“近零空白 + 已知近似位置”的增量场景里很有效。

__whitespace 作为资源__——经典布图把空白（whitespace）当作要消灭的浪费；现代布图把它当**资源**：布线拥塞、缓冲器插入、时序裕量都需要预留空白。固定轮廓本身就把“空白量”变成了由轮廓给定的约束——轮廓紧就只求塞进去，轮廓松就顺带优化线长等其他目标。

__各种“感知（aware）”式布图__——把更贴近实现的指标塞进 SA 代价函数：

- __拥塞感知（congestion-aware）__：把布图切成网格 tile，估每条网穿过 tile 边界的概率，累加成**概率图（probability map）**或用粗全局布线 / 最大并发流估拥塞，把溢出量加进代价；
- __缓冲规划（buffer planning）__：长线必须插缓冲器降延迟，而缓冲器只能放在有空白处，于是布图与缓冲规划要协同（详见缓冲章）；
- __总线驱动（bus-driven）__：要求一条总线穿过的几个宏块在 y（或 x）区间上有足够重叠，让一条固定宽度的带能穿过；
- __微架构 / 吞吐感知__：长线流水（wire pipelining，在线上插触发器）虽降低周期时间但增大块间延迟、抬高 CPI。综合指标是**吞吐率 throughput = 频率 / CPI**；多数方法在 Parquet 代价里塞一个近似吞吐 / IPC 的项；
- __电源 / 地协同（power/ground cosynthesis）__：高功耗块电流大，若离电源焊盘远，长线高阻叠加大电流会恶化 IR 压降，所以把**高功耗块约束在靠近电源焊盘的边界**，并把电源网格密度 / 电源完整性惩罚加进 SA 代价（Liu–Chang，用 B\*-tree）。

还有面向专用架构的变体：FPGA（异构资源向量）、模拟（对称约束，如 ASF-B\*-tree）、3D IC、统计 / 可制造性布图。这些大多是把前述表示 + SA 框架按各自约束改造。

### 6.5 工业布图与原型

工业界说的 floorplanning 和学术界差别很大。学术 floorplanning 多指纯 block 设计的自动摆放，目标是紧致打包 + 最小线长；**工业 floorplanning 允许甚至鼓励人工介入，主目标是时序与可布线性**，面积 / 线长只是手段，还被电源网格、多电压、缓冲插入、时钟树易做性等实际问题主导。

__floorplanning vs prototyping__——两者是芯片在不同抽象层的近似版本：

- __floorplanning（早期）__：更抽象——芯片轮廓、硬块位置朝向、IO 摆放、标准单元可放置区、粗略电源网格。它用在设计可塑、改 RTL 还便宜的早期，核心任务是**诊断**（哪里布线难、时序难、电源够不够）、**容忍不完整 / 不一致数据**（缺块、缺约束甚至缺工艺信息都得能跑，靠设计者填估算值）、**欢迎人工介入**；
- __prototyping（后期）__：floorplan 之上再加一遍快速但完整的“试走一遍”——详细布局 + 粗布线 + 抽取 + 时序，作为开工生产级实现前的**可行性闸门**，证明 RTL 可实现。它对缺数据的容忍度远低于 floorplanning。

__flat vs hierarchical__——平坦（flat）布图把整片当一个问题、任意单元可放任意处，最灵活，但大设计会撑爆工具与人脑、难分工。层次化（hierarchical）把芯片切成 block，每个 block 独立实现，约束（尤其时序、功耗）要**预算（budget）切到各 block**。代价在于：**一个不可行的预算（数十万引脚里的某一个）就可能拖垮整片**，而同样的设计 flat 做反而能完成。理论上层次化并不比 flat 低效（把 flat 结果“切饼干”就得到等效的层次设计），低效来自层次化**工具**的局限，而非层次本身。

__pin assignment + timing budgeting__——切块之后，每个 block 要成自包含单元：每个引脚定位置和层（**pin assignment**），每个引脚定时序约束（**timing budgeting**）。工业典型做法不是去解组合优化，而是**先做一遍 flat 实例：摆好、粗布线一次，在路线穿过 block 边界处定引脚**——这样定出的引脚天然可行（因为整片至少已布通一次），routability 和时序也被这次 flat 布线考虑进去了。时序预算多基于**零 slack 算法（Zero-Slack Algorithm，ZSA，Hauge 等）**：把整片约束分配成各引脚的到达 / 必需时间；做法常是建一遍完整原型、跑抽取 + 静态时序，得到每个信号的可用时间与必需时间，二者之差是 slack，只要预算落在这个区间内就可行。**一个引脚被分到不可行目标，整个设计可能失败**，所以预算必须极小心。

__routability 分析__——布局再好、布不出线也没用；从布局算的时序预算若布线和预期不符也会大错。所以 floorplanner 必须不亲自布线却能较准地预判布线结果，靠 **routability 分析**：用**全局布线（global route）**或**试布线（trial route）**把各网映到粗网格、估每条边的拥塞，看是否超容。难点是“恰好布不出”的临界点很难预测——路由器能靠 rip-up & reroute 补偿几个过满区，再多就崩。常见做法是产出**拥塞颜色图**让工程师凭经验判断。此外工业 floorplan 还要管电源网络设计、ECO 重放（accounting for changes）等。

## 7. 易混淆点 · 课后自测

### 7.1 floorplan 的目标 ≠ 单纯“面积最小”

- floorplan 要同时减小**面积、关键路径延迟、布线拥塞**，而这三者**天然冲突**：面积压太紧会拥塞，通道留太宽会浪费面积，电源网格做太强又抢走信号布线资源。
- **常考**：“floorplan 的目标是不是把面积做到最小？” → **不是**。一味追求面积最小，往往把利用率推到拥塞、绕不出线、时序救不回的地步；floorplan 是在面积 / 时序 / 拥塞 / 供电可靠性之间做的**第一轮工程取舍**，不是单目标最优化。

### 7.2 core limited vs pad limited（核心受限 vs 焊盘受限）

| | 核心受限（core limited） | 焊盘受限（pad limited） |
| --- | --- | --- |
| 谁决定芯片大小 | 核心逻辑规模、宏单元、布线资源 | IO 数量、焊盘间距（pad pitch）、封装引脚分配 |
| 现象 | 核心塞满，IO 环仍有余量 | 晶粒已不能再小，核心区相对空 |
| 优化方向 | 利用率、宏单元摆放、布线资源 | IO 环、焊盘 / 凸点分配、封装协同 |

> **为什么先判断**：两类设计优化方向相反，定芯片尺寸前先判断属于哪一类，才不会在错误方向上使劲。注意 **IO 不随摩尔定律快速缩小**、面积很贵，所以 IO 数量会反过来撑大晶粒。

### 7.3 total vs effective utilization；为什么不能只看利用率

| | 总利用率（total utilization） | 有效利用率（effective utilization） |
| --- | --- | --- |
| 分子 | Σ标准单元 ＋ Σ宏单元 | Σ标准单元 |
| 分母 | 核心区面积 | 核心区 − 阻挡 − 宏留边 − 禁布区 − … |

- **为什么不能只凭利用率定尺寸**：利用率只是面积估算，**不能证明 floorplan 可实现**。**70% 只是常见起点**；引脚密集（pin-dense）单元周围会局部拥塞，即使全局利用率看起来合理仍可能绕不出线。正确动作是跑**试布线 / 全局布线**估拥塞，再回头调综合、核心区尺寸或布线资源。

### 7.4 placement region / placement blockage / routing blockage：各限制什么

| 机制 | 限制对象 | 限制内容 |
| --- | --- | --- |
| 布局区域（placement region） | 标准单元 | 引导 / 限制单元**放在哪个区域**（软引导 → 围栏，由软到硬） |
| 布局阻挡（placement blockage） | 标准单元 | 某区域**不放单元**（硬 / 软 / 部分，三种强度） |
| 布线阻挡（routing blockage） | 走线 | 某区域 / 某些布线层**不走线** |

> 一句话：region 管“放哪儿”，placement blockage 管“能不能放单元”，routing blockage 管“能不能走线”。

### 7.5 halo vs placement blockage（留边 vs 布局阻挡）

- **核心区别**：**halo 跟随宏单元一起移动**（是宏外围一圈空白，改善引脚可达性，给缓冲器 / 电源 / 信号布线留近宏通道）；**placement blockage 多是固定坐标区域**。
- 之所以要单独的 halo 而不只用阻挡：宏会被摆放器移动，**留边跟着宏走**才能始终护住引脚侧的近宏通道；固定坐标的阻挡做不到这一点。

### 7.6 IR drop vs EM（短期电压 vs 长期可靠性）

| | IR 压降（IR drop） | 电迁移（EM） |
| --- | --- | --- |
| 本质 | ΔV = I·R，供电线上的**瞬时电压**够不够 | 长期电流推动金属原子迁移，金属**长期扛不扛得住** |
| 后果 | 电压低于容差 → 单元延迟增大、时序失败、严重时功能错误 | 开路（空洞 void）/ 短路（桥接 bridging）/ RC 退化 |
| 关注期 | **短期电压** | **长期可靠性** |

> 二者都在 floorplan 早期就被电源焊盘、高功耗宏位置、网格宽度 / 间距 / 过孔决定，**不能只看其一**。

### 7.7 flat vs hierarchical：为什么牵出 timing budget、feedthrough、ILM/ETM

- **何时切块**：设计太大时，扁平流程（flat flow）一次跑完会遇到**运行时间和内存**压力，于是改用层次化把芯片切成模块、各自 P&R，再到顶层做全芯片时序与验证。
- **代价不是消失，而是转移**：切块把“大问题”换成“**边界问题**”——
  - **timing budget 与 pin assignment 强相关**：芯片级约束要映射成模块级约束，预算划在哪、引脚摆在哪个边界，直接决定各模块能否独立收敛；
  - **feedthrough**：信号从分区 A 经 B 到 C，若 B 没有穿通就被迫绕过整个模块；
  - **ILM / ETM**：保留接口时序、隐藏模块内部细节，**加速全芯片时序分析**。
- **常考**：“层次化是不是更省事？” → 运行时间 / 内存 / ECO 更省，但**全芯片收敛更难、边界规划更重**，feedthrough、时序预算、接口模型都要额外维护。

### 7.9 课后自测 · 参考答案

1. 目标是**面积、延迟、拥塞三者平衡**，三者天然冲突；一味压面积会把利用率推到拥塞、绕不出线、时序救不回。floorplan 是第一轮工程取舍，不是单目标最优化。
2. **核心受限**由逻辑规模 / 宏 / 布线资源决定大小，优化利用率、宏摆放、布线资源；**焊盘受限**由 IO 数量 / 焊盘间距 / 封装决定晶粒已不能再小，优化 IO 环、焊盘 / 凸点分配、封装协同。
3. IO 单元和焊盘**面积贵、不随工艺快速缩小**；IO 数量多时会反过来把晶粒撑大（焊盘受限），核心逻辑再缩小也没用。
4. 利用率只是**面积估算**，不反映布线拥塞；引脚密集单元会局部拥塞，全局利用率合理仍可能绕不出线——必须跑试布线 / 全局布线验证。
5. **总利用率**分母是核心区面积；**有效利用率**分母还要扣掉阻挡、宏留边、禁布区等，只算真正可放标准单元的面积。
6. 为了给标准单元留**一个完整的大矩形**（摆放器偏好），避免宏把核心区切碎导致拥塞 / 时序变差；引线键合下高功耗宏靠边还能靠近电源焊盘降 IR。
7. 宏（尤其高功耗宏）的**位置决定供电路径长短与电流密度分布**，直接影响 IR 压降和 EM；宏摆放要和电源焊盘、网格密度一起迭代。
8. **placement region** 引导 / 限制标准单元放在哪个区域；**placement blockage** 限制某区域不放标准单元；**routing blockage** 限制某区域 / 某些层不走线。
9. **halo 跟随宏单元移动**（贴身留边，护住引脚可达性与近宏通道）；**placement blockage 多是固定坐标区域**。
10. 解决**信号跨越中间模块**的布线：A→B→C 中 B 没有穿通会被迫绕过整个模块；在 B 上生成穿通引脚 / 穿通网络把连接拆段。
11. 芯片级约束要**映射成模块级约束**（预算划到模块边界），而引脚摆在哪个边界、怎么分配，直接决定边界约束与模块能否独立收敛。
12. 它们**保留接口时序、隐藏模块内部细节**，用抽象模型替代完整模块，使全芯片时序分析更快、更可控。
13. **IR = 短期 / 瞬时电压**够不够（ΔV=I·R，掉太多则单元延迟增大、时序失败）；**EM = 长期可靠性**（电流长期推动金属原子迁移，开路 / 短路 / RC 退化）。
14. 电源线越宽越多越能降 IR / EM，但会**抢占信号布线轨道、加剧拥塞**；要在 IR/EM 与布线资源间取舍，不能把信号资源吃光。
15. 多电压 / 电源门控会关断部分域以省电，但有些逻辑在其他域关断时**仍必须持续供电**；floorplan 阶段（见 1.4「always-on 资源如何预留」、4.6 电源门控 / 多电压）就要预留 always-on 的电源域 / 网络与区域。
16. 物理优化要**按实例独立**移动 / 插缓冲 / 优化；若两实例共享同一模块定义，优化一个会同时改到另一个。唯一化让每个子模块只被引用一次，物理边界才清晰。
17. **（拓展，正文未覆盖，见命令速查）floorPlan = 从零新建布图**（指定尺寸 / 长宽比 / 利用率）；**defIn = 读入已有 DEF**（导入已存在的布图 / 位置）。一个“创建”、一个“读取已有”，混用会覆盖或丢失既有布图。
18.（算法）同一切片布图可对应**多棵**切片树、从而多个 Polish 串，使搜索空间无谓放大且分布不均。规范化只用**斜树（skewed tree）**——任何节点与其右孩子不同标号，对应的 Polish 表达式**没有连续的 V 或连续的 H**，长度 2n−1 且与切片结构**一一对应**（不重不漏）。这正是 Wong–Liu 选规范 Polish 作 SA 状态的原因。
19.（算法）形状曲线是按宽递减、高递增排好的 (宽,高) Pareto 阶梯。**H 割（上下叠）**沿 y 方向合并：`(max(wᵢ,wⱼ), hᵢ+hⱼ)`（高相加、宽取大）；**V 割（左右排）**沿 x 方向合并：宽相加、高取大。合并像归并两个有序表、丢弃被支配的冗余点，自底向上做到根；在根曲线上取面积最小点即全局最优（Stockmeyer O(n²)，Shi 优化到 O(m log m) 并证明是下界）。
20.（算法）两个序列的先后联合定关系：bᵢ 在 Π⁺、Π⁻ 中**都**在 bⱼ 前 → 左于；Π⁺ 后、Π⁻ 前 → 上方。据“左于”建**水平约束图**（点权=宽）、据“下于”建**垂直约束图**（点权=高），各跑一次**最长路**：x 坐标 = 水平图源到该点最长路，y 坐标同理；芯片宽 / 高 = 源到汇最长路。复杂度 O(n²)（LCS 可更快）。
21.（算法）**根 = 左下角的模块**；**左孩子 = 父的右邻紧贴块**（`xⱼ = xᵢ + wᵢ`，同行往右）；**右孩子 = 与父同 x、在其上方的块**（`xⱼ = xᵢ`，往上叠）。DFS + 轮廓线（contour）可 O(n) 还原布局；B\*-tree 与紧致布局一一对应，且 ≡ O-tree。
22.（算法）经典 floorplanning 是**最小面积优化**（无轮廓，在长宽比约束下压面积）；固定轮廓是**可行性问题**——轮廓宽高是输入约束，要把块同时满足宽和高地塞进去（二维约束，不是面积够就行）。**slack** = 块往左下压 vs 往右上压的位置差，衡量可挪余地；零 slack 块在关键路径上，策略是把零-slack 块挪到高-slack 块旁以腾出缩小某维度的空间（Adya–Markov / Parquet）。

> 速查提示：定尺寸前先判**核心受限 / 焊盘受限**；利用率只是估算、**必须试布线验证**；区域 / 阻挡 / 留边分清“放哪儿 / 能不能放 / 能不能走线”；IR 看短期电压、EM 看长期可靠性；层次化把大问题换成边界问题（预算 / 引脚 / 穿通 / 接口模型）。
